🧠 The throughline: you rarely build a model — you adapt one. Pre-trained + a little of your data + the right technique = a custom model in minutes, not months.

The practical reality

You don’t train a transformer from scratch

You now understand the architecture — but training one from zero is out of reach for almost everyone. The practitioner’s move is to download a pre-trained giant and adapt it.

💰 Pre-training from scratch

  • Trillions of words of text
  • Thousands of GPUs, weeks to months
  • Millions of dollars & rare expertise
  • Done by a few labs — not you

🤗 Adapting a pre-trained model

  • Download a released model (often free)
  • A laptop / single GPU, minutes to hours
  • A little of your data
  • Done by everyone — including you, today

🌟 Stand on giants. Someone spent millions teaching a model language; you spend minutes teaching it your task. That leverage is what this module is about.

The core idea · two phases

Pre-training, then fine-tuning

The principle

Transfer learning — reuse, don’t relearn

  • The idea: knowledge learned on a broad task transfers to a narrow one — you adapt, not restart.
  • Human analogy: a fluent English speaker learns legal English fast — they keep the language, add the specialty.
  • Why it works: lower layers learned general meaning that carries over; only the top adapts to your task.
  • The payoff: strong results from a tiny labelled dataset — the model isn’t starting from zero.

🎓 A pre-trained model already “speaks”

It arrives knowing grammar, meaning & world facts. Fine-tuning just teaches it your dialect:

  • …your domain (legal, medical, finance)
  • …your labels (positive / negative, spam / ham)
  • …your style (your brand’s voice)

A few hundred examples can be enough — vs millions to train from scratch.

💡 Fine-tuning is transfer learning in action: keep the giant’s general knowledge, nudge it toward your task with a small, focused dataset.

The fastest on-ramp

Use a pre-trained model in three lines

# Sentiment with a pre-trained model from transformers import pipeline clf = pipeline("sentiment-analysis") clf("I love this course!") # [{'label': 'POSITIVE', # 'score': 0.9997}]

Concrete path 1 · decoders

GPT for text generation

✏️ How generation works

  • A decoder predicts the next token, then repeats — text grows left to right
  • Steer it with the prompt + knobs: max_new_tokens, temperature (randomness)
  • Same idea at every scale — distilgpt2 on a laptop, or GPT-4 in the cloud

🔌 In your labs

  • Generate text with a small local model (distilgpt2) — no key, no cost
  • Tune temperature & length; see how the prompt steers output
  • Optional, guarded: bring your own OpenAI / Groq key for a hosted model
  • → The GPT text-generation lab (Module 3)

💡 The OpenAI GPT API is just a hosted, paid decoder. The mechanism — predict the next token — is identical to the free local model you’ll run, so the skill transfers directly.

Concrete path 2 · encoders

BERT for sentiment classification

📚 How classification works

  • A BERT encoder reads the whole text (both directions) into a rich representation
  • Add a small classification head on top (positive / negative)
  • Fine-tune on labelled examples — the head + a nudge to the encoder

🔬 In your labs

  • Fine-tune a tiny BERT (bert-tiny / distilbert) on a small labelled set
  • Measure accuracy before vs after — watch it learn your task
  • Predict sentiment on brand-new sentences
  • → The BERT sentiment fine-tuning capstone (Module 4)

🧠 GPT generates, BERT understands. Sentiment is an understanding task — so we reach for an encoder, and fine-tune it on a few labelled reviews.

Fine-tuning, end to end

The step that decides success

Preparing a fine-tuning dataset

  • Shape: examples are (text, label) pairs — e.g. a review and positive/negative.
  • Balance: roughly even classes, so the model can’t cheat by always guessing one.
  • Split: hold out a validation set — Day 1’s rule: judge on data it never trained on.
  • Quality > quantity: a few hundred clean, consistent labels beat thousands of noisy ones.
# a tiny labelled dataset texts = ["I loved it", "awful, avoid", "best ever", "waste of time"] labels = [1, 0, 1, 0] # 1=pos 0=neg from sklearn.model_selection import train_test_split X_tr, X_val, y_tr, y_val = train_test_split( texts, labels, test_size=0.25) # now: tokenize -> fine-tune -> evaluate

📝 The labs hand you a small, clean dataset so you can focus on the mechanics — in real work, building this dataset is most of the effort.

Shorter than you’d think

Fine-tuning in a few lines (Hugging Face)

# Fine-tune a small BERT for sentiment (the shape -- the lab has the runnable version) from transformers import AutoTokenizer, AutoModelForSequenceClassification, Trainer, TrainingArguments tok = AutoTokenizer.from_pretrained("prajjwal1/bert-tiny") model = AutoModelForSequenceClassification.from_pretrained("prajjwal1/bert-tiny", num_labels=2) # base + fresh head enc = tok(texts, truncation=True, padding=True) # text -> ids trainer = Trainer(model, TrainingArguments("out", num_train_epochs=3), train_dataset=ds) trainer.train() # the Day-1 loop, handled for you # accuracy before: ~0.50 -> after: ~0.90 the model learned YOUR task

👉 Load (base + head) → tokenize → Trainer.train(). A handful of lines, and the training loop, batching & optimisation are handled. Your Module 4 capstone runs exactly this on a tiny model.

The engineering judgement

Cost, latency & data tradeoffs

ConsiderationHosted API (prompt)Small fine-tuned local model
Time to shipFastest — just call itSlower — gather data & train
Cost at scalePay per call — adds upCheap per call once trained
Data privacyData leaves your wallsStays in-house
Latency & controlNetwork round-tripLocal, fast, fully yours
Upfront effortMinimalLabelled data + training
 No universal winner — match the approach to your constraints: privacy, cost at scale, latency, and how much labelled data you can get.
