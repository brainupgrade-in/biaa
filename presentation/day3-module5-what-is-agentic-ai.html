🧠 The throughline: a language model is a brilliant brain with no hands. Agentic AI gives it hands (tools), a notebook (memory) and a job (goal) — so it can do things, not just describe them.

Let’s define it▶ HANDS-ON · Lab 5.1

What is an AI agent?

An AI agent is a system that uses a language model to decide and take actions toward a goal — in a loop, using tools and memory — with minimal step-by-step human instruction.

🎯 Toward a goal

You give the destination (“answer this”, “draft that”) — not every step.

🔄 In a loop

It can take several steps, checking results as it goes.

🔧 With tools

It can reach the world — search, APIs, code, databases.

🧠 With memory

It remembers — the current task, and facts across tasks.

💡 A chatbot answers once and stops. An agent keeps going — deciding, acting and observing — until the goal is met.

The shift this module is about

A model that answers → an agent that acts

💬 A single LLM call

  • Input → output, once — then it forgets
  • Stateless: no memory of earlier turns
  • No access to the world — only what it was trained on
  • Ask “today’s gold price?” → it guesses from stale data

🤖 An agent

  • Goal → a loop of plan / act / observe
  • Stateful: carries a scratchpad & memory
  • Calls tools — a live price API, a calculator
  • Ask “today’s gold price?” → it looks it up and answers for real

🧠 Same brain, new powers. The agent’s reasoner is the transformer from Days 1–2 — we didn’t make it smarter, we gave it a loop and hands.

A spectrum, not a switch

What makes something “agentic”?

Don’t reach for an agent too soon

Agent vs workflow

🔗 Workflow

  • You write the steps in advance — a fixed sequence
  • The LLM fills in pieces; the path is hard-coded
  • Cheaper, faster, more predictable
  • Best when the steps are known ahead of time

🤖 Agent

  • The model decides the path at run time
  • Chooses which tool, how many steps, when to stop
  • More flexible — and less predictable
  • Best when the path can’t be known in advance

⚖️ Rule of thumb: if you can write the steps down, build a workflow. Use an agent only when the steps genuinely depend on what the model discovers along the way.

The unlock▶ HANDS-ON · Lab 5.2

Tools turn an LLM into an agent

A model alone only produces text. A tool is a function you expose to it — with a name, a description, and inputs — so it can reach beyond words and act.

🔎 Web search

Current facts the model never saw in training — today’s news, prices.

🔢 Calculator

Exact arithmetic — LLMs are bad at math; offload it to a tool.

💻 Code / APIs

Run code, call a weather or maps API, hit your internal services.

🗃️ Databases

Query your own data — orders, tickets, records.

✉️ Actions

Send an email, file a ticket, update a record — change the world.

🔗 …anything callable

If you can wrap it in a function, the agent can use it.

💡 A tool turns “I think the answer is about…” into “I checked, and the answer is…” — the difference between a guess and a fact.

How the model “calls” a tool▶ HANDS-ON · Lab 5.5

Tools & function-calling — a ReAct trace

Question: What is the population of France, divided by 2? Thought: I need the current population — I should search. Action: web_search("population of France 2026") Observation: About 68,000,000 people. Thought: Now I must divide that by 2. I’ll use the calculator (exact math). Action: calculator("68000000 / 2") Observation: 34000000 Thought: I now have everything I need. Final Answer: France’s population (~68M) divided by 2 is about 34 million.

👉 The model doesn’t run the tool — it emits an Action; the orchestrator runs it and pastes back an Observation. That alternation is ReAct — and it’s exactly the trace you’ll read scroll past in your labs.

🛠️ ReAct vs function-calling. The text Thought / Action / Observation above is the readable ReAct rendering. Modern models emit that Action as a structured (JSON) tool call at the API level — function-calling — so there’s no text to hand-parse. Same loop, cleaner mechanism.

So it doesn’t start from zero each step▶ HANDS-ON · Lab 5.6

Memory — short-term & long-term

📝 Short-term · scratchpad

  • The context window — this task’s running transcript
  • Goal + every Thought / Action / Observation so far
  • How the agent knows what it already tried
  • Finite, and clears between sessions

🗃️ Long-term · external store

  • Saved outside the model — often a vector store
  • Facts, notes, past chats — retrieved by meaning
  • Remembers a user / project across tasks
  • The bridge to RAG — coming up on Day 3

🧠 Analogy: short-term memory is what’s on your desk right now; long-term memory is the filing cabinet. A capable agent needs both.

How agents organise their steps▶ HANDS-ON · Labs 5.10 & 5.11

Planning patterns at a glance

🔄 ReAct

Interleave reason & act, step by step — reason, call a tool, observe, repeat.

Simple, robust — the default & where everyone starts.

📋 Plan-and-execute

Draft the full plan up front, then execute the steps — re-plan if reality differs.

Better for long, structured tasks.

🔍 Reflection

The agent critiques its own output and tries to improve it before answering.

Higher quality — at the cost of extra calls.

🧠 “Agent” isn’t one algorithm — it’s a small toolbox of control patterns. Start with ReAct; reach for the others when the task demands it.

The balanced view · the wins

Where autonomous agents genuinely help

🧩 Multi-step tasks

The path isn’t fixed — research, gather, synthesise across several steps.

🔨 Tool orchestration

Combine search + a database + a calculation into one answer.

🔍 Research & automation

Monitor, fetch, summarise, file — the repetitive knowledge work.

🧯 Glue work

Connect systems that don’t talk to each other — the in-between tasks.

🧠 The common thread: the task needs several steps and decisions about which step is next. That’s exactly what a loop-plus-tools is for — and the heart of Day 4’s task automation.

…and where they don’t (be honest)

The risks — agents aren’t free

💰 Cost & latency

  • Many LLM calls per task — cost adds up
  • Each loop step is a round-trip — slow

⚠️ Looping & bad actions

  • Can get stuck repeating an action, burning tokens
  • Hallucinated actions — wrong tool, invented inputs
  • More steps = more chances to go wrong

🛡️ The fix isn’t to avoid agents — it’s to constrain them: cap steps, validate tool inputs, add guardrails and a human checkpoint for risky actions. For a one-shot task, a plain call is cheaper & more reliable — don’t use an agent to do a chatbot’s job.

Make autonomy safe▶ HANDS-ON · Labs 5.8 & 5.9

Guardrails & human-in-the-loop

🚫 Step limits

A max-iteration cap — so the loop can’t run forever.

🔐 Tool permissions

Allow-list tools; read-only by default; gate anything that changes the world.

✅ Validation

Check tool inputs and final outputs before they’re trusted.

👤 Human-in-the-loop

Pause for approval before irreversible, high-stakes actions.

📊 Observability

Log every thought / action / observation — debug & audit the run.

🎯 Right-size autonomy

Turn the dial up only as far as you can safely supervise.

🛡️ We go deep on responsible & safe agents on Day 5 — but build these reflexes from your very first agent.

One query, end to end▶ HANDS-ON · Lab 5.12 (capstone)

A day in the life of an agent

User: “What’s the weather in Tokyo right now, and is that warmer than the June average?”

1 · Reason → Act

“I need the current temp.” → calls weather(Tokyo) → observes 24°C.

2 · Reason → Act

“Now the June average.” → calls search → observes ~22°C.

3 · Reason → Act

“Compare them.” → calculator(24 > 22) → true.

4 · Stop & answer

“It’s 24°C in Tokyo — about 2° warmer than the June norm.”

👉 That’s the loop from earlier — reason → act → observe, twice over — with two tools. You’ll build this loop by hand in the Module 5 labs; Module 6 then hands the same loop to LangChain calling live Google Search & Wolfram Alpha.

Illustrative · preview of Module 6 & your labs▶ BRIDGE · Module 6

A tool-using agent, in code (the shape)

# The SHAPE of a LangChain agent -- Module 6 & the labs have the runnable version from langchain.agents import create_agent from langchain_core.tools import tool # 1. Define a tool: a function + a name + a description the model reads @tool def calculator(expression: str) -> str: """Evaluate a math expression, e.g. '68000000 / 2'.""" return str(safe_eval(expression)) # AST-based -- no bare eval (see the lab) tools = [calculator, web_search] # 2. give the agent its hands # 3. Bind the LLM to the tools and run the loop toward a goal agent = create_agent(llm, tools) # a ReAct agent (a LangGraph app) agent.invoke( {"messages": [("user", "Population of France divided by 2?")]}, config={"recursion_limit": 6}, # a guardrail -- cap the loop ) # > Thought ... Action: web_search ... Observation ... Final Answer: ~34 million

👉 Define tools → create_agent → invoke. Note recursion_limit — a guardrail from the last slide, in one line. Everything you saw today maps to a few lines you’ll write tomorrow.

What “ReAct” stands for

The name of the core agent loop pattern.

AReacting quickly to user input with low latency.
BRecursive Activation of the neural network.
CRetrieve and Activate — a memory technique.
DReason + Act — interleaving thinking with tool actions. ✓

WHY  ReAct = Reason + Act. The agent alternates a Thought with an Action (a tool call), and each Observation feeds the next thought.
