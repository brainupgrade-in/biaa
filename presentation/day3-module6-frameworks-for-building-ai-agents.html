🧠 The throughline: a framework doesn’t make agents smarter — it makes them buildable. It handles the loop, the tools, the memory and the tracing, so you write intent, not plumbing.

You already wrote the loop by hand

Why not just hand-roll it?

🔨 Hand-rolled (Module 5 labs)

  • You parse the model’s text, match a tool, run it, format the result
  • You manage the scratchpad, step counter and stop logic
  • Great to learn — brittle string-glue to maintain
  • Re-invent memory, retries, streaming, tracing yourself

🧰 With a framework

  • Robust output parsing & a tool registry, done for you
  • Ready-made memory, prompt templates, retries, streaming
  • Built-in tracing hooks — see every step
  • You write intent; the framework runs the plumbing

💡 Same move as using a web framework instead of raw sockets. Hand-roll once to understand it — you did — then let the framework carry the boilerplate so your code expresses the agent, not the parser.

The six things every framework provides

What a framework gives you

🔌 Model wrappers

One interface over many providers — swap Ollama ↔ Groq ↔ OpenAI in a line.

🔧 Tool abstraction

Wrap any function as a tool with a name, description & schema the model reads.

🔄 Loop / orchestration

The agent executor that runs reason→act→observe and caps the steps.

🧬 Memory

Ready objects for the scratchpad and conversation history.

🔗 Integrations

Hundreds of pre-built tools & connectors — search, DBs, vector stores.

📊 Observability

Tracing & callbacks — see every thought, action & observation.

🧠 These six are your checklist for evaluating any agent framework — not just the ones we’ll name today.

Don’t be scared of the zoo of names

The agent-framework landscape

The framework your labs use

Meet LangChain

The most widely used open-source toolkit for building LLM apps & agents — composable building blocks (models, prompts, tools, memory) plus an agent runtime that wires them into the ReAct loop.

🧩 Composable

Lego for agents — snap together models, prompts, tools & memory.

🔌 Model-agnostic

Same code runs local Ollama and cloud Groq — a one-line swap.

📦 Huge ecosystem

Hundreds of pre-built tool & data integrations out of the box.

👉 Packages you’ll import: langchain-core (the primitives), langchain (agents & chains), and providers like langchain-ollama / langchain-groq — that’s why the lab imports look the way they do.

Five nouns that make the code readable

LangChain’s core building blocks

🧠 Chat model

The wrapped LLM — ChatOllama, ChatGroq. You .invoke() it.

📝 Prompt template

A reusable prompt with slots — incl. an optional system prompt that steers the agent.

🔧 Tool

A function exposed to the model — usually via the @tool decorator.

🤖 Agent

The reasoning core: given model + tools, it decides the next action — built with create_agent.

⚙️ Agent runtime

The compiled graph create_agent returns — runs the loop, executes tools & caps the steps.

🧬 Memory

The scratchpad & chat history objects, wired into the loop.

👉 Map to Module 5’s anatomy: model = brain, tool = hands, runtime = orchestrator, prompt = instructions, memory = scratchpad. Learn these five nouns and any LangChain snippet reads like a sentence.

The most important thing you’ll write

Defining a tool — @tool

from langchain_core.tools import tool @tool def get_weather(city: str) -> str: """Get the current temperature for a city, in Celsius. Use this whenever the user asks about current weather.""" # ← the MODEL reads this return weather_api.current(city) # return value = the Observation # The model sees THREE things and uses them to decide: # name -> get_weather (keep it clear & verb-like) # docstring -> the description (WHEN to call it -- write it for the model) # signature -> {city: str} (the input schema -- WHAT to pass)

💡 The docstring is not a comment for humans — it’s the instruction the model reads to decide whether to use the tool. A vague docstring is the #1 reason an agent picks the wrong tool. This is Module 5’s “name + description + inputs”, in real code.

The shape you’ll type tomorrow · ≈ Lab 6.1

Assemble an agent — create_agent

from langchain.agents import create_agent from langchain_ollama import ChatOllama # or: from langchain_groq import ChatGroq model = ChatOllama(model="llama3.2:1b") # 1. the brain (swap for ChatGroq) tools = [get_weather, calculator] # 2. the hands agent = create_agent(model, tools) # 3. bind them -> a runnable agent agent.invoke( {"messages": [{"role": "user", "content": "Is Tokyo warmer than 20C right now?"}]}, config={"recursion_limit": 6}, # the loop cap -- a guardrail )

👉 Three moves — model → tools → create_agent — and the framework absorbs all the tool-calling and looping you wrote by hand in Module 5. create_agent returns a runnable agent (a compiled graph); recursion_limit is your guardrail in one line.

What agent.invoke() returns — the trace

The agent loop, made visible

result = agent.invoke({"messages": [...]}) # result["messages"] is the trace HumanMessage Is Tokyo warmer than 20C right now? AIMessage tool_calls=[get_weather(city="Tokyo")] # <- the model DECIDED to call a tool ToolMessage 24 # <- the tools node RAN it AIMessage Yes — it’s about 24°C in Tokyo, warmer than 20°C. # final answer

👉 The agent returns a list of messages — the trace. AIMessage.tool_calls is what the model decided; ToolMessage is what the tools node ran; the last AIMessage is the answer. Every message is a log — this list is your #1 debugging surface.

The concept, now a first-class part

Memory — plugged in, not hand-built

💬 Short-term · conversation

  • Framework keeps the message history & re-injects it each turn
  • “And what about Osaka?” just works — no re-stating context
  • Add it by passing a history / checkpointer — a couple of lines

🗃️ Long-term · vector store

  • Connect Chroma / FAISS as a retriever tool or memory
  • Recall facts across sessions, retrieved by meaning
  • The bridge to RAG — the very next topic on Day 3

🧠 In Module 5 memory was a concept, and in its labs a hand-built scratchpad list. In a framework it’s a component you attach — memory as a part, not a chore.

⚖️ LangChain to get an agent running; LangGraph to get its flow under control. Not rivals — LangGraph is by the same team, same models & tools. The next gear, not a different car. Start simple; graduate when a real need appears.

The fully-autonomous end — and its reality check

AutoGPT & autonomous agents

🚀 The ambition

  • Give a high-level goal — “research the market, write a report”
  • The agent self-prompts: makes its own sub-tasks, picks tools, loops
  • 2023’s viral demo — looked like an autonomous digital worker

⚠️ The reality check

  • Long open-ended autonomy tends to loop & drift off-task
  • Cost piles up; it gets stuck — Module 5’s risks, amplified
  • The field moved toward bounded, structured designs

💡 AutoGPT is an important milestone that taught the field its limits — worth understanding, not the first pattern you reach for. Real systems constrain autonomy: fewer tools, step caps, human checkpoints, graphs. Your capstone will be guardrailed, not unleashed.

Preview of Day 4 · many agents, not one

Multi-agent frameworks

Instead of one agent doing everything, use several specialised agents that collaborate — a researcher, a writer, a reviewer — each with its own role, tools & prompt, passing work between them.

👥 CrewAI

  • Role-based crews: agents with a role, goal & tools
  • A process (sequential / hierarchical) hands off tasks
  • Readable — popular for business workflows

💬 AutoGen (Microsoft)

  • Conversation-based: agents chat to solve a problem
  • Can include a human or a code-executor agent
  • Strong for coding & tool-heavy collaboration

🧠 More agents = better division of labour and more coordination cost & more ways to fail. Same right-sizing discipline as before. We go deep on this — the customer-service chatbot — on Day 4.

Give the agent live capabilities · ≈ Lab 6.2

Connecting to real tools & APIs

🔎 Google Search

  • Live facts beyond the training cutoff — news, prices, events
  • A prebuilt search-API wrapper — often one line

🔢 Wolfram Alpha

  • Exact computation & curated knowledge — math, units, science
  • Precisely where LLMs are weak — offload it

The pattern is always the same three steps:

① get an API key (in .env, never in code)  →  ② wrap the service as a tool (usually a prebuilt integration)  →  ③ add it to the tools list. Now the same agent answers questions no static model could — Module 5’s “day in the life” trace, for real.

You can’t fix what you can’t see

Observability & debugging

1 · verbose=True

Prints the ReAct trace — free, enough to debug your first agents.

2 · Callbacks

Hook every model & tool call — log latency, tokens, errors in code.

3 · Tracing platform

LangSmith / Langfuse — every run as a visual trace you can inspect & score.

🧠 Agents are non-deterministic — so tracing isn’t optional. Once you can see and score runs (inputs, each step, cost, timing), you improve the agent systematically instead of guessing — the difference between “it sometimes works” and “here’s exactly where it broke.”
