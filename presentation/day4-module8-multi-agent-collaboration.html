The ceiling of a soloist

Why one agent isn’t always enough

😕 The overloaded agent

  • One prompt doing everything — roles blur, instructions get forgotten
  • Too many tools — selection accuracy drops as the catalog grows
  • No separation of concerns — a policy tweak breaks unrelated behaviour
  • Can’t work in parallel or bring distinct expertise to bear

🧩 The same fix as software

  • These are the problems modules & single-responsibility solve
  • Decompose: several focused specialists, small prompts, few tools each
  • Each is easier to prompt, test & debug in isolation
  • A coordinator wires them together

💡 A multi-agent system is decomposition applied to agents. But decomposition has a price — reach for it when a single agent is genuinely straining, not by default. We’ll return to that judgement at the end.

A working definition

What is a multi-agent system?

Several agents, each with a focused role, that collaborate to complete a task none would handle as well alone. Three defining properties:

🎯 Specialisation

Each agent has its own prompt, tools & expertise — billing, tech, research.

🔄 Coordination

A pattern wires them — a supervisor routes, a pipeline chains, a fan-out parallelises.

💬 Communication

They pass messages / share state — one agent’s output is another’s input.

🧠 Not just one agent calling tools — here the “tools” are themselves reasoning agents with their own judgement. LangGraph is the natural home: the Module-6 state graph scales up to a whole team — nodes are agents, edges are handoffs.

The building block

Specialist roles: separation of concerns

The payoff comes from each agent being good at one thing. That’s three deliberate choices per agent:

📝 A focused prompt

A short role definition — this job and nothing else. Small prompts get followed.

🔧 A small tool set

Only the tools this role needs — high selection accuracy, no cross-talk.

🚪 Clear boundaries

Knows what’s out of scope and hands back rather than guessing.

💡 Heuristic: give each agent a job description you could hand a new hire. If you can’t write it in three lines, the role is too broad. That’s what lets you improve the billing agent without touching tech support.

The centrepiece · the backbone of the chatbot

The supervisor (router) pattern

The plumbing of collaboration

Message passing & shared state

📩 Message passing

Agents exchange explicit messages — “handle this refund, order 4471” → a reply. A growing transcript.

🗃️ Shared state

All agents read/write a common state — the conversation, fields, findings, the plan. LangGraph’s model.

💡 Two rules keep it sane: keep the shared context small & relevant (don’t dump everything to everyone — that recreates the overloaded prompt and blows up cost), and make handoffs explicit (“done, hand back” / “this needs the tech agent”). The failure mode is agents talking past each other over a state nobody owns.

The four shapes · composable primitives

Orchestration shapes

Shape 1 · the relay

Sequential — a pipeline of specialists

Agents run in a fixed order, each transforming the previous output — researcher → writer → editor. A relay where the baton is the shared state.

✅ When to use

The task has ordered stages, each needing different expertise; stage N depends on N−1.

💡 Why it works

Each agent gets a clean, focused input — and you can swap any stage independently.

⚠️ Watch out

Errors propagate — validate between stages; and it’s serial, so latency adds up.

🧠 It’s the multi-agent version of Module 7’s automation pipeline — same idea, but each stage is a reasoning specialist. Often the right first thing to reach for.

Shape 2 · many views at once

Parallel — fan-out for coverage & speed

Several agents work the same input concurrently; their outputs are then combined. Latency is the slowest agent, not the sum.

🔎 Coverage

Different lenses on one question — security, performance, correctness reviewers — catch what one would miss.

🎯 Diverse attempts

Several agents solve it different ways; pick or merge the best — how you beat one-shot answers.

👉 Fan-out creates a new problem — and it’s the pivot of this module: now you have several outputs and need one. That convergence is decision making, and you must design it: voting, critique, synthesis — next.

The other half of the title

Reaching a decision — the challenge

When several agents contribute, you must converge on one outcome — one answer, one action, one plan. Skip this and a multi-agent system falls apart:

🔄 Endless deferral

Agents defer to each other — “you decide” / “no, you” — and never converge.

🧩 Incoherent merge

Contradictory outputs get concatenated into a mess.

📣 Loudest wins

The last / loudest agent silently wins — with no principle behind it.

🧠 The fix: choose a deliberate decision mechanism up front — voting, critique, synthesis — and for high stakes, a human makes the final call. A multi-agent system without one is just noise with extra steps.

Decision mechanism 1 of 3

Voting & consensus

Several agents give comparable answers; take the agreement. Independent errors cancel — if each is 70% right with uncorrelated mistakes, the majority is far better.

🗳️ Majority & variants

Plain majority · weighted by trust · scored by a judge agent.

👤 Split → escalate

No clear majority is a signal — escalate to a human, don’t force it.

✅ Fits convergent tasks

Facts, classification, checkable answers — not open-ended creative work.

⚠️ Watch for a shared blind spot: agents that reason alike will confidently agree on the same wrong answer — voting amplifies correlated error. Treat a split vote as information, not a failure.

Decision mechanism 2 of 3

Critique & debate

Don’t average — stress-test. One agent proposes; an independent critic tries to find what’s wrong; the proposer revises, or a judge rules.

🔍 Critic

A separate agent with an adversarial brief reviews & revises — catches the author’s blind spots.

⚖️ Debate

Two agents argue opposing sides; a judge decides — surfaces the trade-offs.

🔄 Reflection loop

Propose → critique → improve, capped — or they argue forever.

🧠 Generating and evaluating are different skills — separating them raises quality, just as code review beats writing carefully alone. Use it when being right matters more than being fast; it costs extra calls & latency.

Decision mechanism 3 of 3

Synthesis — combine the parts

Voting is for comparable whole answers; synthesis is for complementary parts. One agent weaves the collected findings into a single coherent response.

🧩 Resolve, don’t paste

Reconcile or flag conflicts — never just concatenate contradictory outputs.

📌 Stay grounded

Built only from what agents actually found — invents nothing (Module 7).

🎤 Own the voice

One consistent tone, though many agents contributed.

🧠 The three, together: vote to converge comparable answers, critique to harden a risky one, synthesise to combine parts. It’s the last step of the chatbot — then a human approves anything risky (draft-not-send, at team level).

The worked example · your Lab 4.2

The multi-agent customer-service chatbot

The run, made visible

The chatbot’s trace

Message: "I was charged twice for order 4471 and the app keeps crashing." Supervisor: two intents — billing + technical. Route to both. ↳ Billing agent lookup_invoice("4471") Observation: 2 charges found -> duplicate. Refund warranted. ↳ Tech agent known_issues("app crash on open") Observation: matches BUG-231 -> fix: update to v4.2. Synthesise: "Hi — I've flagged the duplicate charge on 4471 for refund, and the crash is a known issue fixed in v4.2..." Final: status="needs_approval" # refund is irreversible -- a human approves

👉 The trace shows which agent did what, in what order, and where the human gate is. The team drafted a full resolution to two problems — then stopped before the irreversible refund.

Illustrative · the shape of Lab 4.2

The chatbot, in code

# Each specialist = a Module-6 create_agent: its OWN system_prompt + small tool set def build_specialist(tools, role): return create_agent(llm, tools, system_prompt=f"You are the {role} specialist. Use ONLY your own tools.") AGENTS = {"billing": build_specialist([lookup_invoice], "billing"), "tech": build_specialist([known_issues], "tech")} involved = route(msg) # supervisor = route pattern (M7) -> ["billing","tech"] findings = {name: AGENTS[name].invoke( # each runs with ITS OWN tools... {"messages": [("user", msg)]}, config={"recursion_limit": 8})["messages"][-1].content for name in involved} reply = synthesize(findings) # combine findings -> one DRAFT reply if "billing" in involved and refund_intent(msg): # refund is irreversible -> if human_approves(reply): issue_refund(reply) # draft-not-send, team level

👉 Nothing new here — it’s all Days 3–4 parts arranged as a team: @tool, create_agent, shared state, the route pattern, and the draft-not-send gate. Multi-agent is composition, not a new toolkit.

The senior judgement

When to build a team — and when not to

✅ Reach for multi-agent when

  • The task needs genuinely distinct expertise that bloats one prompt
  • You need parallelism for coverage or speed
  • One agent’s tool list has grown so large selection fails
  • There are clear specialist stages or departments

⛔ The costs you take on

  • More model calls — a team multiplies token bill & latency
  • More failure modes — handoff loops, coordination bugs
  • Harder debugging — spread across agents & shared state
  • A single agent / workflow is your first attempt

🧠 Every agent you add must earn its keep. Anthropic’s reality check: prefer the simplest thing that works; add agents & orchestration only when simpler approaches fall short. Keep it as simple as the problem allows — and no simpler.

New failure modes — make them visible

Failure modes & observability

🔄 Handoff loops

A → B → A forever — cap total handoffs, like recursion_limit.

💬 Talking past

A shared state nobody owns — agents miss or overwrite context.

💰 Cost / latency

Each agent is calls — a chatty team can 10× your bill.

🔍 Lost accountability

Which agent was wrong? Without tracing you can’t tell.

📊 The defence is observability: log every agent, message, handoff, tool call & final decision — so you can replay the conversation, find the faulty agent, and watch cost. LangSmith / Langfuse do this for graphs. A multi-agent system is only as trustworthy as it is observable.

Recap

Five things to carry forward

1 · Decompose into specialists

Reach for a team when one agent is overloaded — focused roles, separation of concerns.

2 · Coordinate with a pattern

Supervisor + pipeline + parallel, over message passing / shared state.

3 · Design the decision

Vote to converge, critique to harden, synthesise to combine.

4 · The chatbot

Supervisor → specialists → synthesise → human approves — your Lab 4.2.

5 · A team must earn its keep

More calls, latency & failure modes than a soloist — add agents only when needed, and make it observable.
