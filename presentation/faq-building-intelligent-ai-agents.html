Day 1 — Foundations of AI & Deep Learning
Day 1 · Module 1

AI & Its Evolution

7 questions
QWhat's the difference between AI, machine learning, and deep learning?▼
Quick answerThey're nested circles, not rivals. AI is the broadest goal (machines doing smart things). ML is one way to get there (learn from data instead of hand-coded rules). Deep learning is one kind of ML (multi-layer neural networks).

AI ⊃ ML ⊃ DL. A hand-written chess rules engine is AI but not ML. A spam filter that learns from examples is ML. A neural net that reads handwriting is deep learning. Deep learning is dominant today because it scales with data and compute where older ML plateaus.

In this course: Module 1 walks the whole progression — rules → data → ML pipeline → a neural net — so you feel where each layer sits.

QIf we can just write rules, why do we need machine learning?▼
Quick answerBecause for many problems nobody can write the rules. How do you write an if-statement that recognises a handwritten "7"? ML lets the computer infer the rules from examples instead.

Rules are great when the logic is known and stable (tax brackets, business validation). They collapse when the problem is fuzzy, high-dimensional, or constantly changing (images, language, fraud). ML trades "a human writes the rules" for "a human provides labelled examples and the model finds the pattern."

In this course: the first labs literally start rule-based, hit the wall, then switch to a data-driven model — you experience why the switch happens.

QWhat does it actually mean for a model to "learn"? Isn't it just memorising?▼
Quick answerLearning = adjusting internal numbers (weights) to reduce error on examples, in a way that generalises to data it has never seen. Pure memorising is the failure mode — it's called overfitting.

Training shows the model examples, measures how wrong it is (loss), and nudges the weights to be less wrong. The goal isn't to ace the training data — it's to do well on new data. A model that only memorised would score perfectly on what it saw and fail on everything else.

QWhat is overfitting — why is a model that scores 100% on training data a problem?▼
Quick answerOverfitting is when a model memorises the training data (including its noise) instead of learning the general pattern. It scores great on what it saw and poorly on new data — the only score that matters.

Analogy: a student who memorises the answer key vs. one who understands the subject. The first aces the practice test and fails the real exam. That's why we always hold out a validation/test set the model never trains on — the gap between training and validation accuracy reveals overfitting.

In this course: a Module 1 lab deliberately overfits so you can see the train-vs-test gap open up; Module 2 adds early stopping to fight it.

QIs a neural network the same as how the human brain works?▼
Quick answerNo — it's loosely brain-inspired, not a model of the brain. Artificial "neurons" are simple math (weighted sum + activation). Real neurons are vastly more complex, and the brain doesn't learn by backpropagation.

The metaphor is useful for intuition and stops there. Treat "neuron" and "network" as convenient names for math, not biology — it prevents a lot of confusion later.

QWhy does the AI timeline (slide 8) start at 1950? Didn't Turing break Enigma in the 1940s?▼
Quick answerBecause the timeline is a history of AI, not of computing. Turing's Enigma Bombe (1939–45) was codebreaking hardware — no learning, no reasoning. 1950 is his paper Computing Machinery and Intelligence (“Can machines think?” + the Turing Test) — the birth of AI as an idea.

Turing sits at the head of both stories: 1936 the Turing machine (theory of computing) → 1939–45 the Bombe (codebreaking) → 1950 “Can machines think?” (AI is born as an idea) → 1956 the Dartmouth workshop coins the term “Artificial Intelligence.” The Bombe was a fast, special-purpose search machine, not a thinking one — it belongs on a computing timeline, not this one.

QIf hand-written rules (expert systems) count as AI, why isn't the Enigma Bombe AI? It also just follows rules.▼
Quick answerAI is classified by the task it attempts and the tradition it comes from, not the mechanism. Not every rule-follower is AI (a thermostat isn't). An expert system reasons over encoded knowledge to do an intelligence-needing task (diagnosis); the Bombe brute-force searches cipher settings.

Three tests separate them: (1) is the task one that needs human-like intelligence? — diagnosis yes, cipher search no; (2) does it represent knowledge and infer new conclusions? — expert system yes, Bombe no; (3) which research lineage? — the AI program vs. cryptanalysis.

The counter that sharpens it: Deep Blue is basically brute-force search too, yet it's AI — because chess is a canonical intelligence challenge, it encodes chess knowledge, and it came from the AI community. So mechanism alone never decides it.

Honest note: “AI” has no crisp, mechanism-based definition; by a strict behavioural view you could file the Bombe as narrow AI. The “AI effect” (“AI is whatever hasn't been done yet”) means the label keeps moving — which is exactly what the “What is AI?” and “Rules vs learning” slides are meant to provoke.

Day 1 · Module 2

Introduction to Deep Learning

8 questions
QWhat is a "neuron" in a neural network?▼
Quick answerA neuron is a tiny function: it takes inputs, multiplies each by a weight, adds them up plus a bias, and passes the result through an activation function. That's it.
output = activation( w1·x1 + w2·x2 + ... + b )

The weights and bias are the numbers learned during training. A "deep" network is just many of these stacked in layers, so the output of one layer feeds the next.

In this course: Module 2's first labs build a single neuron by hand in NumPy before stacking them.

QWhy do we need activation functions? What breaks without them?▼
Quick answerThey add non-linearity. Without them, stacking 100 layers is mathematically identical to one layer — the network could only ever draw straight lines.

A weighted sum is linear. Chain linear things together and you still get a linear thing. Activations (ReLU, sigmoid, tanh) bend the output, letting the network model curves, corners, and complex patterns. ReLU (max(0, x)) is the modern default because it's simple and trains fast.

QWhat is a loss function and why do we need one?▼
Quick answerA loss function is a single number that measures how wrong the model is. Training is nothing more than making that number smaller.

Without a loss you can't tell "better" from "worse", so there's nothing to improve toward. Regression often uses MSE (mean squared error); classification uses cross-entropy. The loss is the target the whole training loop optimises.

In this course: Module 2 has a dedicated MSE lab, then uses it to drive gradient descent.

QWhat is gradient descent — how does the model actually improve?▼
Quick answerGradient descent is "which way is downhill, take a small step, repeat". The gradient says how the loss changes as you nudge each weight; you step the weights in the direction that lowers loss.

Analogy: fog on a hillside, you can't see the bottom, but you can feel the slope under your feet and step downhill. The learning rate is how big each step is. Backpropagation is the efficient algorithm that computes those slopes for every weight at once.

QWhat are epochs, batches, and the learning rate?▼
Quick answerEpoch = one full pass over the training data. Batch = the chunk of examples processed before each weight update. Learning rate = how big each update step is.
  • Learning rate too high → overshoots, loss bounces or explodes.
  • Too low → trains painfully slowly, may get stuck.
  • Too few epochs → underfit; too many → overfit.

These are hyperparameters — you set them, the model doesn't learn them. Tuning them is a core skill; a Module 2 lab has you sweep them and read the curves.

QWhy doesn't adding more layers/neurons always help?▼
Quick answerA bigger network has more capacity to memorise — so past a point it overfits, trains slower, and needs more data. Bigger is only better if you have the data (and regularisation) to match.

Capacity is a budget: too little and the model underfits (can't capture the pattern); too much and it overfits (captures noise). The right size is found empirically by watching the validation curve, not by assuming "more is better."

QDo the weights update after every batch, or after every epoch?▼
Quick answerAfter every batch, not every epoch. One batch through the loop = one weight update. An epoch is one full pass over the data — made of many batches, so many updates.

The loop runs per batch: forward → loss → backward → update → next batch. Example: 10,000 images with batch size 100 → 100 batches → 100 updates per epoch. This is mini-batch gradient descent, the standard — hundreds of small corrective nudges per epoch instead of one, which learns faster and is more stable. (Update-once-per-epoch is full-batch GD; batch size 1 is stochastic GD.)

QWhy are adam and sparse_categorical_crossentropy used together?▼
Quick answerThey're not a pair — they're two independent choices on the same compile() line. The loss fits the problem type; the optimizer is a general-purpose default. You pick them for unrelated reasons.

Loss answers "how wrong?": sparse_categorical_crossentropy is for multi-class classification with integer labels (MNIST gives labels as 3, not one-hot) — same math as categorical_crossentropy, just a different label format. Optimizer answers "how do I step the weights?": adam is the robust default that works out of the box for almost any problem. Swap the loss to binary_crossentropy or mse and adam wouldn't change — that's the tell they're independent.

Day 2 · Module 3

Why Transformers?

9 questions
QWhat is tokenization — why not just feed the model words or letters?▼
Quick answerTokenization splits text into subword pieces (roughly 4 characters each), each with an integer ID. It's a compromise between whole words (vocabulary explodes, can't handle new words) and single letters (sequences get too long, meaning too thin).

Subwords let a model represent "tokenization" as token + ization — so it handles rare or unseen words by composing pieces, while keeping the vocabulary manageable (~50k–100k tokens). Everything downstream (embeddings, attention) operates on these token IDs.

In this course: Module 3 has both a tokenization lab and a subword-tokenizer lab.

QWhat are embeddings?▼
Quick answerEmbeddings turn things (words, sentences, images) into vectors of numbers, arranged so that similar things sit close together and different things sit far apart. Meaning becomes geometry.

A computer can't work with the word "king" directly, so we map it to a list of numbers (e.g. 300 or 768 of them). The numbers aren't random — they're learned so that relationships are encoded as positions in space:

  • "king" and "queen" land near each other; "king" and "banana" land far apart.
  • Relationships become arithmetic: king - man + woman ≈ queen.

Why it matters: raw words are discrete and meaningless to math. Embeddings give a dense, meaningful space where you can measure similarity (usually cosine similarity — the angle between vectors), do semantic search, and feed vectors into models.

def cosine_similarity(a, b): return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)) # ~1.0 = very similar, ~0 = unrelated, negative = opposite

In this course: Module 3 labs build embeddings + cosine similarity by hand (offline NumPy), and semantic search with TF-IDF. The same idea powers ChromaDB + RAG in the later agent modules.

QWhat do embeddings mean inside an LLM?▼
Quick answerAn LLM only ever computes on vectors. Text is turned into token embeddings, refined layer by layer into contextual embeddings, then converted back into a probability over the next token.

1. Token embeddings — the entry point

Text → tokens (subwords) → integer token IDs → a lookup in a learnable embedding matrix of shape [vocab_size, d_model] (one vector per token in the vocabulary).

2. Positional information

"dog bites man" ≠ "man bites dog", so the model adds positional encoding (sinusoidal, or modern RoPE) so each vector knows what the token is and where it sits.

3. Contextual embeddings — the key insight

Old static embeddings (Word2Vec) gave "bank" one fixed vector forever. In an LLM, attention lets each token look at the others and update itself — so "bank" in "river bank" ends up near water, and in "bank loan" near money. Same token, different vector, depending on context.

4. Output — back to tokens

The final vector is multiplied by the vocabulary matrix → a logit per token → softmax → P(next token).

Two things called "embeddings"Internal token embeddings flow through the LLM to generate text. Embedding models (e.g. text-embedding-3, sentence-transformers) output one vector per sentence/document — that's what you store in a vector DB for RAG.
QHow does the attention step actually work?▼
Quick answerEach token asks a Query, compares it against every token's Key to get scores, softmaxes the scores into weights, then takes a weighted sum of Values. That blend becomes its new, context-aware vector.

Every token vector is projected into three vectors with learned matrices:

  • Query (Q) — "what am I looking for?"
  • Key (K) — "what do I advertise?"
  • Value (V) — "what info do I pass on?"

Library analogy: your Query is the topic you want, each book's Key is its spine label, its Value is the content inside. Match query to keys, then collect content from the best matches.

The four steps (for one token)

  • 1. Score: dot Q with every K → relevance scores.
  • 2. Scale: divide by √d_k for numerical stability.
  • 3. Softmax: turn scores into weights that sum to 1.
  • 4. Weighted sum: blend the Values by those weights → the new vector.
Attention(Q, K, V) = softmax( Q·Kᵀ / √d_k ) · V

Multi-head attention: run many of these in parallel, each learning a different relationship (grammatical subject, adjacency, long-range reference), then concatenate.

Causal mask (GPT-style): future tokens' scores are set to −∞ before softmax, so a token can only attend to itself and the past. That one change is the difference between BERT (bidirectional) and GPT (left-to-right).

In this course: Module 3's "attention-by-hand" lab computes these four steps in NumPy; the "attention heatmap" lab plots the softmax weights.

QCan you show a tiny worked numerical example of attention?▼
Quick answerSentence "cat sat mat", 2-D vectors. We compute the new vector for "cat" and get [6.69, 3.31] — a blend of the Values "cat" chose to attend to.

Setup (Q for "cat" is [2, 0]; d_k = 2, so √d_k ≈ 1.414):

TokenKey (K)Value (V)
cat[1, 0][10, 0]
sat[0, 1][0, 10]
mat[1, 1][5, 5]
# Step 1 - Score: Q·K for each token Q·K_cat = [2,0]·[1,0] = 2 Q·K_sat = [2,0]·[0,1] = 0 Q·K_mat = [2,0]·[1,1] = 2 # raw scores [2, 0, 2] # Step 2 - Scale by √d_k = 1.414 [2, 0, 2] / 1.414 = [1.414, 0, 1.414] # Step 3 - Softmax -> weights (sum to 1) e^1.414=4.11 e^0=1.00 e^1.414=4.11 sum=9.22 weights = [0.446, 0.108, 0.446] # Step 4 - Weighted sum of Values x: 0.446×10 + 0.108×0 + 0.446×5 = 6.69 y: 0.446×0 + 0.108×10 + 0.446×5 = 3.31 new_cat ≈ [6.69, 3.31]

What to notice: equal-scoring tokens (cat, mat) get equal weight; the irrelevant one (sat) is suppressed but not zeroed — softmax is soft. Change the Query and the whole output changes: the query is "what am I looking for." And the √d_k divisor stops large dimensions from making softmax too sharp (which would kill gradients).

QWhat is positional encoding, and why is it needed if attention sees everything?▼
Quick answerAttention is order-blind by default — it treats the input as a bag of tokens. Positional encoding injects "where each token sits" into its vector so word order carries meaning.

Without it, "dog bites man" and "man bites dog" would look identical to the model. The original transformer added fixed sinusoidal patterns; modern LLMs often use RoPE (rotary) which encodes relative position and generalises better to longer inputs.

In this course: Module 3 has a positional-encoding lab so you see the patterns added to the embeddings.

QWhy did transformers replace RNNs/LSTMs? What was wrong with them?▼
Quick answerRNNs read text one token at a time, in order — slow to train (no parallelism) and forgetful over long distances. Transformers process the whole sequence at once and let any token reach any other in a single step.
  • Parallelism: RNNs are sequential by design; transformers compute all positions simultaneously → far better GPU use → train on much more data.
  • Long-range memory: RNN signal degrades over distance; attention connects distant tokens directly.

That combination is exactly what made today's large models possible.

QWho published the paper on transformers?▼
Quick answer"Attention Is All You Need" (2017), by eight authors primarily at Google (Brain & Research), presented at NeurIPS 2017. arXiv: 1706.03762.

The eight authors: Ashish Vaswani, Noam Shazeer, Niki Parmar, Jakob Uszkoreit, Llion Jones, Aidan N. Gomez (Univ. of Toronto), Łukasz Kaiser, Illia Polosukhin. A footnote states contribution was equal and author order was randomized.

Why it's a landmark: it dropped recurrence (RNNs/LSTMs) and convolutions entirely, relying only on self-attention — making training massively parallel and scalable. That directly enabled BERT, GPT, T5 and everything since. 100,000+ citations.

Trivia: several authors went on to found major AI efforts — Aidan Gomez co-founded Cohere; Noam Shazeer co-founded Character.AI and later co-led Google Gemini; Illia Polosukhin co-founded NEAR.

  • Vaswani et al. (2017) — Attention Is All You Need
QBERT doesn't "transform" anything — why is it called a Transformer?▼
Quick answer"Transformer" is the name of an architecture, not a description of turning text into other text. The "transform" happens internally: it transforms input vectors into richer vectors, layer by layer — which BERT does just as much as GPT.

Asking "but BERT doesn't transform anything" is a bit like asking "why is it called Java if it's not coffee?" — the name is a proper noun, not a verb about the output.

There is a real transform, just not the one you're picturing. Recall the worked example: "cat" went in as [2, 0] and came out as [6.69, 3.31]. Each layer transforms every token's representation into a more context-aware one. BERT does exactly this — it just uses the result to understand (classify, embed) rather than to generate new words.

Why it's legitimately in the family: BERT is built from transformer components — self-attention (Q/K/V), stacked transformer blocks, positional encoding. It uses the encoder half (bidirectional); GPT uses the decoder half (causal). The name spells it out: Bidirectional Encoder Representations from Transformers — the "T" is Transformer.

One-liner: "Transformer" names the architecture, not the task. It transforms vectors → richer vectors. BERT does this to understand; GPT does this to generate.

Pre-trained Models & Fine-tuning

5 questions
QAre BERT and GPT "algorithms"?▼
Quick answerNot quite. BERT and GPT are model architectures / families of trained models — a design + a training objective + the resulting weights. The algorithms are what build them: gradient descent, backpropagation, the attention computation.

The layered vocabulary

AlgorithmA step-by-step procedureBackprop, gradient descent, Adam
ArchitectureThe shape of the networkTransformer, CNN, RNN
Model (family)Architecture + training objectiveBERT, GPT, T5

Loosely, people say "the GPT algorithm" in casual speech — not worth fighting over, but strictly it's a model architecture trained by algorithms.

In this course: the "fine-tune BERT for sentiment" labs use BERT because it's an understanding/encoder model; the text-generation labs use a GPT-style decoder. Choosing between them is the encoder-vs-decoder decision above.

QWhat is fine-tuning, and why not just train a model from scratch?▼
Quick answerFine-tuning takes a model that already learned language from billions of examples and nudges it toward your specific task with a small dataset. Training from scratch would need enormous data, compute, and money to re-learn what's already known.

The pretrained model already understands grammar, meaning, and context. You're just adapting that general knowledge — like hiring a fluent speaker and teaching them your company's jargon, rather than teaching a baby to talk. This is a form of transfer learning.

In this course: Module 4's advanced labs fine-tune a sentiment classifier and show the before-vs-after jump in accuracy.

QPrompt engineering vs fine-tuning vs RAG — when do I use which?▼
Quick answerThree ways to steer a model, cheapest first: prompt (change the instructions), RAG (give it the right documents at question time), fine-tune (change the model's weights).
ApproachChangesBest for
PromptingJust the input textBehaviour/format tweaks; fast, free, first thing to try
RAGAdds retrieved contextGiving the model knowledge it lacks (your docs, fresh facts)
Fine-tuningThe model's weightsA consistent skill/style/format at scale; specialised classification

Rule of thumb: try prompting first, reach for RAG when the problem is missing knowledge, fine-tune when it's a missing skill you need reliably and repeatedly. They combine — a fine-tuned model can still use RAG.

QWhat is transfer learning / a "frozen" model?▼
Quick answerTransfer learning = reuse a model trained on one task as the starting point for another. "Freezing" means locking most of the pretrained weights and training only a small new head on top.

The frozen body acts as a fixed feature extractor (it already knows how to represent language/images); the small trainable head learns your specific labels. This needs far less data and compute than full fine-tuning and resists overfitting.

In this course: Module 4's offline-graded labs do exactly this — frozen TF-IDF features + a trainable classifier head, the portable form of fine-tuning.

QWhat's the difference between a base model and an instruct/chat model?▼
Quick answerA base model just predicts the next token — it completes text but doesn't reliably follow instructions. An instruct/chat model is a base model further trained (instruction tuning + RLHF) to follow requests and hold a conversation.

Ask a base model "What is the capital of France?" and it might continue with more quiz questions instead of answering. The instruct version answers. Almost everything you use as an assistant (ChatGPT, Claude) is the instruct-tuned kind.

Day 3 · Module 5

What Is Agentic AI?

6 questions
QWhat's the difference between an LLM and an agent? Isn't ChatGPT already an agent?▼
Quick answerAn LLM answers in one shot: text in → text out. An agent wraps the LLM in a goal, a loop, tools, and memory so it can take actions, observe results, and keep going until the task is done.

The LLM is the brain; the agent is the brain plus hands and a to-do loop. Plain ChatGPT can only talk. An agent can call a calculator, search the web, read a file, check the result, and decide the next step. The difference is acting in a loop toward a goal, not just responding.

In this course: Module 5's first lab is literally "model vs agent"; you then build the loop from scratch.

QWhat is the ReAct loop (reason → act → observe)?▼
Quick answerReAct is the core agent cycle: the model Reasons about what to do, Acts by calling a tool, Observes the tool's result, then loops — until it has the answer.
Thought: I need today's exchange rate. Action: get_rate("USD","INR") Observation: 83.2 Thought: Now I can compute the total. ... loop until done ...

The loop is what turns a one-shot predictor into something that can chain steps and recover from surprises. In this course: Module 5 has you build this loop in plain Python, driven by a deterministic "mock LLM" policy.

QWhat is a "tool" for an agent — is it just an API call?▼
Quick answerA tool is any function the agent is allowed to call — a calculator, a database query, a web search, an email sender. It's how the agent affects or reads the world beyond generating text.

Each tool has a name, a description (so the model knows when to use it), and typed inputs/outputs. An API call is one common kind of tool, but a local function or DB lookup counts too. Which tools you give an agent defines what it can — and can't — do, which is also your main safety lever.

QWhat stops an agent from running forever or going rogue?▼
Quick answerGuardrails — explicit limits you code around the loop: a max-steps cap, a tool allow-list, loop detection, and input validation. The agent's autonomy is bounded by design, not by hope.
  • Max steps: the loop can't run more than N iterations.
  • Allow-list: it can only call tools you explicitly approved.
  • Loop detection: repeating the same action/observation halts it.
  • Input validation: reject malformed or dangerous tool arguments.

In this course: Module 5 has a dedicated guardrails lab; the theme returns hard in Modules 9–10.

QWhat's the difference between plan-and-execute and ReAct?▼
Quick answerReAct decides the next step one at a time as it goes. Plan-and-execute writes the whole plan up front, then carries it out step by step.

ReAct is flexible and adapts to surprises but can wander. Plan-and-execute is more structured and predictable for complex multi-step tasks, but a bad initial plan hurts. Many real agents combine them: plan, execute, and re-plan if reality diverges. Reflection/self-critique (a Module 5 lab) adds a step where the agent checks its own work.

QDoes the agent actually "decide", or is it just the LLM predicting text?▼
Quick answerMechanically it's still next-token prediction — but wrapped in a loop that executes those predictions as actions, the effect is decision-making. The "decision" is the model choosing which tool-call text to emit next.

It's useful to hold both truths: there's no little person inside, just prediction; and the loop turns those predictions into real, consequential actions. That's exactly why guardrails matter — a confident wrong prediction becomes a confident wrong action.

Day 3 · Module 6

Frameworks for Building AI Agents

6 questions
QWhy do I need LangChain — can't I just call the LLM API myself?▼
Quick answerYou can — and in Module 5 you did. A framework like LangChain just saves you from re-writing the plumbing: the agent loop, tool routing, memory, prompt templating, retries, and tracing. You write intent; it runs the machinery.

For a one-off script, raw API calls are fine. As soon as you want tools, memory, multi-step loops, and observability, you're rebuilding a framework by hand. LangChain gives you those as composable pieces. The tradeoff is a layer of abstraction to learn — which is why the course builds the loop from scratch first, so the framework feels like a shortcut, not magic.

QWhat's the difference between LangChain and LangGraph?▼
Quick answerLangChain is great for linear "chain of steps" agents. LangGraph models the agent as a graph of nodes with state — better for loops, branches, human-in-the-loop pauses, and multi-agent flows.

Reach for LangGraph when your agent needs cycles, conditional routing, or a human approval step in the middle — things awkward to express as a straight chain. LangChain is the simpler default; LangGraph is where you graduate when control flow gets complex.

In this course: Module 6 shows a LangGraph-style state graph with a human-in-the-loop node.

QWhat is the AgentExecutor / the loop the framework runs?▼
Quick answerThe AgentExecutor is the framework's run-the-ReAct-loop-for-you engine: it asks the model what to do, calls the chosen tool, feeds the result back, and repeats — up to max_iterations.

It's the exact loop you hand-built in Module 5, now provided and hardened. max_iterations is a built-in guardrail (the max-steps cap). Understanding it means the framework is never a black box — you know precisely what it's doing each turn.

QWhat's the @tool decorator actually doing?▼
Quick answerIt registers a plain Python function as a tool the agent can call — taking the function's name, docstring, and signature and turning them into the description the model reads to decide when to use it.
@tool def get_weather(city: str) -> str: """Return the current weather for a city.""" ...

The docstring matters — it's how the model knows what the tool does. A vague description means the agent uses the tool badly. In this course: Module 6's first lab is writing your first @tool.

QIs AutoGPT the same kind of thing as LangChain?▼
Quick answerRelated but different. LangChain is a toolkit/library you build agents with. AutoGPT is a ready-made autonomous agent that tries to self-direct toward a goal with minimal steering.

LangChain gives you control and building blocks. AutoGPT is more "set a goal and let it run" — impressive as a demo, but harder to make reliable and safe for real work. The course leans on LangChain because control and guardrails matter for production agents.

QWhat is a PromptTemplate — why not just use an f-string?▼
Quick answerA PromptTemplate is a reusable prompt with named slots. You can — for something trivial — use an f-string; the template adds validation, reuse, composability, and integration with the rest of the framework.

It keeps prompts as first-class, testable objects rather than strings scattered through code, and plugs cleanly into chains and agents. For quick experiments an f-string is fine; for a maintained agent, templates pay off.

Day 4 — Real-World Agents
Day 4 · Module 7

Task Automation

6 questions
QHow is an automation agent different from a normal script or cron job?▼
Quick answerA script follows fixed steps you wrote. An agent decides steps at run time using an LLM — it can read messy input, choose tools, adapt, and handle cases you didn't hard-code.

Use a script when the process is fixed and known. Use an agent when the input is unstructured or the path varies (e.g. "read this email and draft an appropriate reply"). The cost is less determinism — which is why agent automation leans hard on validation and human approval.

In this course: Module 7 builds the email-drafting agent piece by piece as an automation pipeline: trigger → gather → draft → validate → approve → act.

QWhy should the email agent draft but not send?▼
Quick answerBecause sending is irreversible and outward-facing. Draft-not-send keeps a human in the loop for the consequential action — the agent does the work, a person approves the send.

The pattern: give the agent all the gather tools it needs, but withhold send_email. It returns a needs_approval draft instead of acting. You get the productivity of automation without the risk of an agent confidently emailing the wrong thing to a customer.

In this course: this is Module 7's explicit human-in-the-loop gate; the capstone never auto-sends.

QWhat is structured output and why does it matter?▼
Quick answerStructured output means the model returns a predictable shape (like JSON with named fields) instead of free-form prose — so your code can reliably use it without fragile text parsing.

"The customer wants a refund of $40 for order 123" is hard to act on programmatically. {"intent":"refund","amount":40,"order":"123"} is trivial to route and validate. Structured output is the glue that lets an LLM plug into real systems.

QWhat are the extract and route patterns?▼
Quick answerExtract = pull structured fields out of messy input. Route = decide which path/handler/tool the input should go to. Together they turn raw input into a typed decision.

A support message comes in → extract intent + details → route to the billing flow vs the tech flow. These two patterns show up in almost every real automation agent; Module 7 has dedicated labs for each and chains them in the capstone.

QWhat is idempotency and why does an agent need it?▼
Quick answerIdempotent = doing the same action twice has the same effect as doing it once. Agents retry on failure, so without it a retry could send two emails or issue two refunds.

You get it with things like unique request keys or "check before acting". Because agents loop and retry, idempotency is what makes retries safe instead of dangerous. Module 7 pairs it with a retry lab.

QHow do I stop the agent from doing the wrong action confidently?▼
Quick answerThree layers: validate the draft/action before it runs, withhold the dangerous tools, and gate the irreversible step behind a human. Confidence is not correctness.

An LLM will state a wrong answer as fluently as a right one. So you never trust the output blindly — you check it against rules (validation), limit what it can do (least privilege), and require approval for anything you can't undo.

Day 4 · Module 8

Multi-Agent Collaboration

6 questions
QWhy use multiple agents instead of one big smart agent?▼
Quick answerSeparation of concerns. Specialist agents (billing, tech, legal) each with a focused prompt and toolset are easier to build, test, and keep reliable than one giant agent trying to do everything.

Same reason software has modules and teams have roles: a focused agent has a smaller, clearer job, fewer tools to misuse, and is easier to debug. A supervisor routes each request to the right specialist. The cost is coordination overhead — which is the rest of this module.

In this course: Module 8 builds the customer-service chatbot as a team of specialists behind a supervisor.

QWhat is a supervisor / router agent?▼
Quick answerThe supervisor is the traffic controller: it reads the request, decides which specialist(s) should handle it, and routes it there — then optionally synthesises their outputs into one reply.

It's the multi-agent version of the route pattern from Module 7, but the destinations are whole agents rather than functions. It also handles multi-intent requests ("I have a billing question and a bug") by dispatching to more than one specialist.

QWhat's the difference between sequential and parallel agent execution?▼
Quick answerSequential = a pipeline; each agent's output feeds the next. Parallel (fan-out) = several agents work at once on independent parts, then results are combined.

Use sequential when steps depend on each other (research → draft → edit). Use parallel when parts are independent (get billing info and tech info simultaneously) — it's faster. A supervisor can do both: fan out, then synthesise.

QWhat is a "handoff" and why cap it?▼
Quick answerA handoff is one agent passing control to another. You cap the number of handoffs so agents can't ping-pong forever ("this isn't mine" → "not mine either" → ...).

Uncapped handoffs are a classic multi-agent failure — a loop that burns tokens and never resolves. An explicit cap (plus loop detection) is the multi-agent guardrail, the same discipline as max-steps for a single agent.

QHow do agents "agree" — voting, debate, synthesis?▼
Quick answerThree common patterns: voting (run several, take the majority), critique/debate (agents challenge each other's answers in a capped loop), and synthesis (one agent merges multiple outputs into a single grounded reply).

Voting improves reliability on factual calls. Debate surfaces weaknesses one agent alone would miss. Synthesis produces the final, coherent answer. Each has a lab in Module 8; the debate loop is capped so it can't run forever.

QDoesn't more agents just mean more cost and more ways to fail?▼
Quick answerYes — multi-agent is not free. It costs more tokens and adds coordination failure modes. Use it when a task genuinely benefits from specialisation or independent perspectives, not by default.

A single well-prompted agent is often enough. Reach for multi-agent when the work splits cleanly into roles, or when independent voices materially improve quality (voting/debate). Always add observability and loop-detection — more moving parts need more visibility.

Day 5 · Module 9

Agents in Industry (High-Stakes)

6 questions
QWhy can't a finance agent just give investment advice?▼
Quick answerBecause investment advice is regulated, liability-bearing, and consequential. The agent is built to be informational only — it grounds and explains figures; a licensed human decides.

This isn't a limitation to work around — it's the design. A no-advice guardrail keeps the agent from saying "buy" or "sell", and the trade tool is deliberately withheld. It informs; it doesn't act or advise.

In this course: Module 9's financial-report insight agent embodies this throughout — it's the responsible-AI point of the module.

QWhat does "grounding" and "citing every claim" mean for an agent?▼
Quick answerGrounding = every figure the agent states is tied back to its source in the actual document. Citing = it shows that source alongside each claim, so a human can verify.

This is the antidote to hallucination in high-stakes settings. If the agent says "revenue was ₹42 crore", it must point to the line it read that from. No source → not stated. Grounding + citations make the output auditable, not just plausible.

QWhy withhold the trade tool — isn't automation the whole point?▼
Quick answerAutomating insight is valuable; automating an irreversible, high-consequence action (placing a trade) is where you stop. The agent gets read-only tools; a human makes the call.

This is least privilege applied to finance: give the agent exactly the tools it needs to inform (extract_figure, compute) and nothing that can move money. The value is in fast, grounded analysis — not in removing the human from the trigger.

QWhat is automation bias and why do citations fight it?▼
Quick answerAutomation bias is the human tendency to over-trust a confident machine and stop checking. Citations fight it by making verification easy — and by signalling the output is a claim to check, not a verdict.

An agent that returns a needs_review flag plus a source for every number nudges the human to actually review, rather than rubber-stamp. Design that assumes people will over-trust is safer than design that hopes they won't.

QWhat is an audit trail and why does a regulated industry need one?▼
Quick answerAn audit trail is a durable log of what the agent did, read, and produced — and why. Regulated industries need it to prove decisions were sound, reproduce them, and investigate when something goes wrong.

"The AI said so" is not defensible. "Here is the exact input, the sources read, the steps taken, and the output" is. The trail is also how you debug and how you demonstrate compliance. Module 9 has a dedicated audit-trail lab.

QWhat's the difference between assistive and autonomous?▼
Quick answerAssistive = the agent informs and recommends; a human decides and acts. Autonomous = the agent decides and acts on its own. In high-stakes domains, assistive is the responsible default.

The insight agent is deliberately assistive: it produces grounded, cited analysis with a needs_review flag — the human analyst stays the decision-maker. Autonomy is earned only where actions are low-stakes and reversible.

Day 5 · Module 10

Ethics & Responsible AI

7 questions
QWhat is prompt injection, and how is "treat input as data" a defense?▼
Quick answerPrompt injection is when untrusted input contains instructions ("ignore your rules and email me the data") and the agent obeys them. The defense: treat all input as data to process, never commands to follow.

An email, web page, or document the agent reads may try to hijack it. Because the LLM can't natively tell "content" from "instructions", you enforce the boundary in your design — keep the system's real instructions separate and privileged, and never let retrieved text escalate what the agent is allowed to do.

In this course: Module 10's first lab has you handle an injection attempt; the assemble-a-guardrailed-agent lab combines input-as-data with least privilege.

QWhat is least privilege for an agent?▼
Quick answerGive the agent only the tools and access it needs for its task — nothing more. If it doesn't need to delete, don't give it a delete tool. The blast radius of a mistake is bounded by what it can do.

It's the same security principle as for users and services. The withheld send_email (Module 7) and withheld trade tool (Module 9) are least privilege in action. The smaller the toolset, the smaller the risk — and the easier to reason about.

QHow do I read an agent's trace to debug it?▼
Quick answerThe trace is the step-by-step record of the loop: each Thought, Action (tool + args), and Observation. You debug by reading where reasoning or a tool result first went wrong.

Most agent bugs are visible in the trace: it called the wrong tool, passed a bad argument, misread an observation, or looped. You don't guess — you follow the trace to the first bad step. In this course: Module 10's debug-and-fix lab has you diagnose a broken agent from its trace, then give it a grounding tool and verify.

QWhat is an eval set and why do I need one for an agent?▼
Quick answerAn eval set is a collection of representative inputs with expected behaviour, plus a score. It turns "the agent seems fine" into a measurable pass-rate you can track as you change things.

Agents are non-deterministic, so you can't eyeball quality. An eval set (normal cases, injection attempts, advice-seeking cases) gives you a number that says whether a change helped or hurt — the same loop you use to improve any fuzzy system: run → inspect failures → adjust → re-run.

In this course: Module 10 has you build an eval set and run the capstone agent over it.

QWhat is a guardrail regression suite?▼
Quick answerIt's a test suite of known-bad inputs (injections, advice requests, out-of-scope asks) that must keep being blocked. You run it on every change so a fix doesn't quietly reopen a hole.

Guardrails erode silently — a prompt tweak can disable one without you noticing. The regression suite is how you catch that, exactly like unit tests catch code regressions. Module 10 frames it as a deploy gate: fail the suite, don't ship.

QHow do I detect a runaway loop?▼
Quick answerWatch for repetition (same action/observation again and again) and enforce a step cap. If the agent isn't making progress toward the goal, halt it.

Signs: identical tool calls, cycling between two states, step count climbing with no closer answer. The fixes are the guardrails from Module 5 — max-steps, loop-detection — now used as a diagnostic. Module 10 has you classify the failure mode from the trace.

QWhat does "fairness across groups" mean for an agent?▼
Quick answerIt means the agent's behaviour and quality shouldn't systematically differ across groups of people (by gender, region, language, etc.) in ways that cause harm. You check by measuring outcomes per group, not just overall.

An agent can score well on average yet fail one group badly — the average hides it. Fairness work means slicing your eval by group and looking for gaps. Module 10 has a fairness-across-groups lab that makes this concrete.
