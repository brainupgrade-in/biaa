From “an agent that acts” to “an agent that does a job”

💬 A demo agent

  • You type a question; it reasons and answers
  • Runs when you’re watching
  • “Works” means it worked once
  • Great for learning & showing the loop

⚙️ An automation agent

  • A real event triggers it — a client email arrives
  • Pulls context from your tools & data, produces a work product
  • “Works” means it works every time, unattended
  • Checks itself & pauses for a human before irreversible acts

🧠 Same loop, more rigour. The model and the ReAct loop are unchanged from Day 3 — what we add is the engineering that makes an agent dependable enough to leave running.

A working definition

What is agentic task automation?

Using an agent to complete a recurring business task end to end — from a trigger to a finished work product — using tools for context and guardrails for safety.

💬 More than a chatbot

A chatbot converses; an automation delivers a result and often takes an action.

🧰 More than a script

Classic RPA breaks on anything unstructured; an agent handles the fuzzy middle.

🎯 The sweet spot

Repetitive enough to automate, too unstructured for a rigid script.

💡 The agent’s LLM brain is what lets it read a messy email, decide the right reply, or pull fields from free text — the parts a plain script could never handle. That’s a huge slice of everyday knowledge work.

The centrepiece · the shape of every automation

The task-automation pipeline

Gather before you draft

Grounding the task in real data

A general model doesn’t know your client’s history, your refund policy, or today’s numbers. So the agent must gather context first — then draft. Three staple sources:

🔧 Tools & APIs

Live systems — the CRM, ticketing, calendar, a payments API.

🗃️ Databases

Your records — orders, customers, transactions.

📚 Knowledge base

Policies, templates, product docs — often via RAG retrieval.

💡 Gather first, draft second. An agent that drafts before it gathers hallucinates specifics; one that grounds every claim in retrieved context is accurate and auditable. These are just the Module-6 tools, pointed at a real job.

The backbone of reliable automation

Structured output, not prose

💬 Prose (for a human)

“I think we should refund them, and it seems fairly urgent — something about order 4471.”

Readable — but your code can’t reliably act on it.

🔢 Structured (for a machine)

{ "intent": "refund",
  "urgency": "high",
  "order_id": 4471 }

Machine-readable, validatable, composable.

🧠 Ask the model for a defined shape (JSON / a schema), not a paragraph — frameworks do this with function-calling / structured output. The mantra: prose for humans, JSON for machines. If code will act on it, make it structured.

Almost every automation is one of three

Three patterns: Draft · Extract · Route

Pattern 1 of 3 · the email lab

Draft — generate a work product

The agent turns gathered context into a draft a human will review — a reply, a summary, a proposal. Three keys:

📌 Ground it

Feed the real context — the message, the history, the policy — so it’s specific, not generic.

🎤 Give it a voice

A system prompt sets tone, format & limits — “concise, never promise a refund.”

👤 Keep the human

A draft is a suggestion, not a sent message — the human edits & approves.

⚠️ The failure mode: a draft that invents a delivery date or a policy because it wasn’t grounded. Draft agents are high-value and low-risk — the human still holds the pen. That’s why the email agent is the canonical first real-world lab.

Pattern 2 of 3

Extract — mess in, structure out

Pull defined fields out of unstructured input — the workhorse that turns email, chat, docs & forms into rows your systems can process.

# unstructured in "Hi, my order from last Tuesday still hasn't arrived, ref 4471, getting frustrated." # structured out (a tight schema: fixed fields, allowed values) { "order_id": 4471, "intent": "delivery_issue", "sentiment": "negative" }

🧠 Keys: a tight schema (only fields you’ll use, with allowed values) · handle missing data (return null, don’t invent an id) · validate before trusting. Often the first step in a chain — extract → route → draft.

Pattern 3 of 3

Route — decide what happens next

Emit a label from a fixed set that drives a branch — which team, how urgent, auto-handle or escalate. The closed list is the trick that makes an LLM a reliable classifier.

📋 Closed label set

One of {billing, tech, sales, other} — include “unsure” as an escape hatch.

👤 Confidence & fallback

Low confidence → route to a human. Never guess on high stakes.

⎇ The label drives code

Your workflow branches on it — it’s a decision, not a note.

👉 Routing makes one agent the front door to a whole system. Routing to the right specialist agent is the bridge to Module 8 — multi-agent collaboration.

What separates a demo from production

Reliability: validate, retry, stay idempotent

✅ Validate the output

Check the JSON parses, required fields exist, values are in range — don’t act if it fails.

🔄 Retry on failure

Models & tools are flaky — retry with a firmer prompt, but cap the attempts.

🔑 Idempotency

Running twice must be safe — key each action so you never double-send or double-charge.

🤝 Degrade gracefully

Can’t finish confidently? Fall back to a human with context — never guess silently.

💡 Assume every step can fail, and make failure safe and visible. This is ordinary software engineering — agents don’t get a pass on it. Idempotency matters most for anything that touches money or messages.

The rule for real-world agents

Human-in-the-loop: draft ≠ send

The worked example · your Lab 4.1

The email-drafting agent, end to end

The run, made visible

The email agent’s trace

Trigger: client email — "Where is my order 4471? It's late." Thought: I need the order status first. Action: lookup_order("4471") Observation: {status: "shipped", eta: "Friday", carrier: "BlueDart"} Thought: I'll ground the reply in our delivery-delay template. Action: get_template("delivery_delay") Observation: "Hi {name}, your order {id} is on its way..." Thought: I have everything. Draft the reply and validate it. Draft: "Hi Priya, your order 4471 has shipped and is due Friday via BlueDart..." Validation: OK -- no invented dates, required fields present. Final: status="needs_approval" # drafted, NOT sent -- waiting for a human

👉 The agent’s output is a draft + a “needs approval” flag — it gathered, drafted and validated autonomously, then stopped before sending. That last line is human-in-the-loop, in the trace.

Illustrative · the shape of Lab 4.1

The email agent, in code

# Tools give the agent its hands (Module 6). Note what's MISSING: send_email. @tool def lookup_order(order_id: str) -> dict: """Look up an order's status, ETA and carrier by id.""" return orders_db.get(order_id) tools = [lookup_order, get_template] # gather-only -- the agent CANNOT send agent = create_react_agent(llm, tools, draft_prompt) executor = AgentExecutor(agent=agent, tools=tools, max_iterations=6) draft = executor.invoke({"input": incoming_email}) # returns a DRAFT, not a sent mail if human_approves(draft): # the human-in-the-loop gate send_email(draft) # a human triggers the irreversible act

👉 The strongest human-in-the-loop guardrail is the simplest: don’t give the agent the send_email tool. It can gather and draft all day — it literally cannot send. A human does that after approving.

Choose the right targets

When to automate — and when not to

✅ Good candidate

  • Repetitive & high-volume — worth the build
  • Rule-ish, but with unstructured input
  • Tolerates a human check (draft-and-approve fits)
  • Clear success — you can tell right from wrong

⛔ Poor candidate

  • Rare or one-off — just do it by hand
  • High-stakes with no room for oversight
  • Needs judgment you can’t specify
  • A wrong answer is catastrophic & uncatchable

🧠 ROI lens: value ≈ volume × time saved × (how cheap & catchable errors are). Automate the boring, frequent, checkable 80% — like email drafting — and leave the rest to people.

Assume it will fail — make failure visible

Failure modes & observability

💩 Hallucinated specifics

An invented date or policy — fixed by gather-first + validation.

🤫 Silent wrong output

Plausible but incorrect — fixed by validation + human approval.

🔌 Tool / data failure

CRM down, stale lookup — fixed by retries & graceful degradation.

📉 Drift

Worked at launch, inputs changed — caught by monitoring.

📊 The common defence is observability: log every run — trigger, each tool call, the draft, the validation, the human decision. That log lets you debug, audit & measure (approval rate, edit rate). LangSmith / Langfuse do this — you can’t run unattended what you can’t see.
